# IMU-Based Drift Detection — MQTT Protocol Reference

## Overview

The Tinkerforge radial drives (axis_b on `cam1` and `cam2`) use step counting
for position feedback but have no physical end-stops. Over time the step
counter can drift from physical reality.

This protocol uses the **onboard IMU** of each OAK-D 4 Pro camera to verify
the true camera angle whenever the drive arrives at a named **checkpoint**
position. If drift exceeds a configurable threshold, the Pi automatically
applies a corrective move and publishes a drift event.

**Only axis_b (Tinkerforge) drives are checked.** GPIO linear drives (axis_a)
do not use this protocol.

---

## System Roles

| Role | Side | Responsibility |
|------|------|----------------|
| **IMU publisher** | Windows (`02_Oak_D_Windows_streaming`) | Reads OAK-D IMU via DepthAI SDK, computes roll + pitch, publishes `IMUAngle` messages |
| **Drift detector** | Pi (`01_Oak_D_MQTT`) | Subscribes to IMU telemetry, requests fresh readings at checkpoints, detects and corrects drift |

---

## MQTT Topics

| Topic | Direction | QoS | Retained | Model |
|-------|-----------|-----|----------|-------|
| `telemetry/cam/{cam_id}/imu` | Win → Pi | 0 | No | `IMUAngle` |
| `cmd/cam/{cam_id}/imu_check` | Pi → Win | 1 | No | `IMUCheckRequest` |
| `event/drift/{cam_id}/{axis}` | Pi → Win | 1 | No | `DriftDetectionEvent` |

`cam_id`: `cam1` or `cam2`  
`axis`: always `b` for this feature

---

## Payload Schemas

### `IMUAngle` — `telemetry/cam/{cam_id}/imu`

Published by Windows continuously (~2 Hz) and immediately in response to an
`IMUCheckRequest` (with `request_id` echoed).

```json
{
  "cam_id": "cam1",
  "roll_deg": -14.73,
  "pitch_deg": 2.15,
  "request_id": null,
  "timestamp": "2026-04-14T10:22:01.123Z"
}
```

| Field | Type | Description |
|-------|------|-------------|
| `cam_id` | string | Camera identifier (`cam1` \| `cam2`) |
| `roll_deg` | float | Camera roll angle in degrees (computed from accelerometer) |
| `pitch_deg` | float | Camera pitch angle in degrees (computed from accelerometer) |
| `request_id` | string \| null | Echoed from `IMUCheckRequest` for correlation; `null` in background 2 Hz publishes |
| `timestamp` | ISO 8601 UTC | Time the reading was taken |

---

### `IMUCheckRequest` — `cmd/cam/{cam_id}/imu_check`

Published by the Pi after a Tinkerforge drive settles at a checkpoint position.
Asks Windows to publish a fresh `IMUAngle` with the matching `request_id`.

```json
{
  "request_id": "a3f7c2d1-4e8b-4f21-9c0e-1b2d3e4f5a6b",
  "cam_id": "cam1",
  "checkpoint_name": "top_view",
  "timestamp": "2026-04-14T10:22:00.000Z"
}
```

| Field | Type | Description |
|-------|------|-------------|
| `request_id` | string (uuid4) | Unique ID generated by Pi; must be echoed in the `IMUAngle` response |
| `cam_id` | string | Camera to read IMU from |
| `checkpoint_name` | string | Name of the checkpoint being verified (for logging) |
| `timestamp` | ISO 8601 UTC | When the request was sent |

---

### `DriftDetectionEvent` — `event/drift/{cam_id}/{axis}`

Published by the Pi when drift is detected and a correction is attempted.

```json
{
  "request_id": "a3f7c2d1-4e8b-4f21-9c0e-1b2d3e4f5a6b",
  "cam_id": "cam1",
  "drive_axis": "b",
  "checkpoint_name": "top_view",
  "expected_angle_deg": -14.73,
  "actual_angle_deg": -17.10,
  "drift_deg": -2.37,
  "correction_steps": 284,
  "corrected": true,
  "timestamp": "2026-04-14T10:22:03.450Z"
}
```

| Field | Type | Description |
|-------|------|-------------|
| `request_id` | string | Echoed from the `IMUCheckRequest` that triggered this check |
| `cam_id` | string | Affected camera |
| `drive_axis` | string | Always `"b"` |
| `checkpoint_name` | string | Name of the checkpoint where drift was detected |
| `expected_angle_deg` | float | Calibrated expected angle at this checkpoint |
| `actual_angle_deg` | float | Measured angle from IMU |
| `drift_deg` | float | `actual − expected` (signed; negative = moved too far) |
| `correction_steps` | int | Steps applied to compensate (`−drift_deg × steps_per_degree`, clamped to ±10°) |
| `corrected` | bool | `true` if the correction move completed successfully |
| `timestamp` | ISO 8601 UTC | When the event was published |

## Request/Response Sequence

```
Pi                          MQTT Broker                 Windows
│                               │                           │
│  Drive arrives at checkpoint  │                           │
│  Settling delay (150 ms)      │                           │
│                               │                           │
│──cmd/cam/cam1/imu_check──────►│──────────────────────────►│
│  (request_id: "abc-123")      │                           │
│                               │   [reads OAK-D IMU]       │
│                               │◄──telemetry/cam/cam1/imu──│
│◄──────────────────────────────│   (request_id: "abc-123") │
│                               │                           │
│  Compare actual vs expected   │                           │
│  Drift = actual − expected    │                           │
│                               │                           │
│  [if |drift| > threshold]     │                           │
│  Apply corrective move        │                           │
│──event/drift/cam1/b──────────►│──────────────────────────►│
│                               │                           │
│  Publish position (reached)   │                           │
│──status/drives/cam1/position─►│──────────────────────────►│
│                               │                           │
```

### Timeout Behaviour

If Windows does not respond to `imu_check` within **3 seconds** (configurable
in `DriftDetector.request_imu_check()`), the Pi logs an error and **continues
the drive sequence without performing a drift check**. The drive is not blocked.

---

## Continuous Background Publishing

In addition to responding to `imu_check` requests, the Windows controller
publishes `telemetry/cam/{cam_id}/imu` continuously at **~2 Hz** for live
monitoring. These background messages have `request_id: null`.

The Pi caches the latest angle for each camera:
- Displayed in real time on the IMU Calibration tab of the GUI
- **Not used** to resolve pending `imu_check` Futures — only messages with
  a matching `request_id` resolve a pending request

## Configuration: `config/imu_calibration.yaml`

```yaml
# Degrees of drift before auto-correction triggers
drift_threshold_deg: 2.0

# Named checkpoints per camera, axis_b only
checkpoints:
  cam1:
    b:
      - name: top_view
        drive_position: 1000         # Tinkerforge step count at this position
        expected_roll_deg: -14.730   # Captured during calibration
        expected_pitch_deg: 2.150
        active_angle: roll           # Which angle to use for drift check: roll | pitch
      - name: side_view
        drive_position: 4500
        expected_roll_deg: 5.210
        expected_pitch_deg: 1.890
        active_angle: roll
  cam2:
    b: []

# Conversion factor: Tinkerforge steps per degree of camera rotation
# Determine by jogging the drive between two positions, measuring:
#   steps_per_degree = (step_delta) / (angle_delta_degrees)
steps_per_degree:
  cam1_b: 120.0
  cam2_b: 120.0
```

## Calibration Workflow (Operator Guide)

### Prerequisites

- Windows controller running and connected to the MQTT broker
- Both OAK-D cameras online (IMU data flowing on `telemetry/cam/+/imu`)
- Navigate to **Calibration** page on the Pi GUI (`:8080/calibration`)

### Step 1 — Determine Steps-per-Degree

1. Jog the radial drive (axis_b) to a known position, note the step count and IMU roll/pitch
2. Jog to a second position, note the new step count and IMU angle
3. Calculate: `steps_per_degree = |Δsteps| / |Δangle_degrees|`
4. Enter the value in the **Steps per degree** field and click **Save**

> Repeat for cam2:b independently — the two drives may have different gear ratios.

### Step 2 — Capture Checkpoints

For each capture position you want to verify:

1. Jog the radial drive to the target position
2. Wait for the live **Roll** and **Pitch** values to stabilise
3. Type a checkpoint name (e.g., `top_view`, `pos_1`)
4. Select the **Active angle** (`roll` or `pitch`) — whichever changes significantly with drive movement
5. Click **Capture Here**

The checkpoint is saved immediately to `config/imu_calibration.yaml`.

### Step 3 — Set Drift Threshold

Adjust the **Drift threshold** (default: 2.0°) based on your accuracy requirements.
Click **Save**.

### Step 4 — Test

Send a `MoveCommand` with `checkpoint_name` set to a calibrated checkpoint name.
Monitor `event/drift/{cam_id}/b` to see if drift events are published.

## Drift Detection Algorithm

```
1. Pi publishes IMUCheckRequest to cmd/cam/{cam_id}/imu_check
2. Pi awaits IMUAngle with matching request_id (timeout: 3s)
3. active = checkpoint["active_angle"]   # "roll" or "pitch"
4. actual  = imu.roll_deg OR imu.pitch_deg
5. expected = checkpoint["expected_{active}_deg"]
6. drift = actual − expected              # signed degrees

7. if |drift| <= drift_threshold_deg:
       return  # within tolerance, no action

8. correction_steps = clamp(−drift × steps_per_degree, −10°_equiv, +10°_equiv)
   # correction is capped at 10° equivalent to protect against IMU noise spikes

9. drive.move_to(current_position + correction_steps, speed=0.5)

10. publish DriftDetectionEvent to event/drift/{cam_id}/b
```

**Correction cap:** If the computed correction exceeds `10 × steps_per_degree`
steps (i.e., more than 10° equivalent), the correction is clamped and a
`CRITICAL` log entry is written. This protects against IMU noise spikes causing
large unintended moves. Verify calibration if this occurs.

## Windows-Side Implementation Notes

The Windows controller (`02_Oak_D_Windows_streaming`) must:

### Subscribe
```
cmd/cam/+/imu_check
```

### On receiving `IMUCheckRequest`
1. Read the IMU from the relevant OAK-D camera via DepthAI IMU pipeline
2. Compute `roll_deg` and `pitch_deg` from the accelerometer reading
3. Publish `IMUAngle` to `telemetry/cam/{cam_id}/imu` with `request_id` echoed

### Background publishing
Publish `IMUAngle` (with `request_id: null`) to `telemetry/cam/{cam_id}/imu`
at ~2 Hz for all connected cameras.

### DepthAI IMU pipeline (reference)
```python
imu = pipeline.create(dai.node.IMU)
imu.enableIMUSensor(dai.IMUSensor.ACCELEROMETER_RAW, 100)
imu.setBatchReportThreshold(1)
imu.setMaxBatchReports(10)

imu_out = pipeline.create(dai.node.XLinkOut)
imu_out.setStreamName("imu")
imu.out.link(imu_out.input)
```

Computing angles from accelerometer:
```python
import math

def compute_angles(accel) -> tuple[float, float]:
    ax, ay, az = accel.x, accel.y, accel.z
    roll_rad  = math.atan2(ay, az)
    pitch_rad = math.atan2(-ax, math.sqrt(ay**2 + az**2))
    return math.degrees(roll_rad), math.degrees(pitch_rad)
```

## Pydantic Models (Pi-side, `src/shared/models.py`)

```python
class IMUAngle(BaseModel):
    cam_id: str
    roll_deg: float
    pitch_deg: float
    request_id: str | None = None
    timestamp: datetime = Field(default_factory=_now)

class IMUCheckRequest(BaseModel):
    request_id: str
    cam_id: str
    checkpoint_name: str
    timestamp: datetime = Field(default_factory=_now)

class DriftDetectionEvent(BaseModel):
    request_id: str
    cam_id: str
    drive_axis: Literal["a", "b"]
    checkpoint_name: str
    expected_angle_deg: float
    actual_angle_deg: float
    drift_deg: float
    correction_steps: int
    corrected: bool
    timestamp: datetime = Field(default_factory=_now)
```

Windows side should maintain its own copies of these models
(or import from a shared package if one is created in future).
